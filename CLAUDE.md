## Coordinate system (robot-car)

**Convention:** −X = front/bumper · +X = rear · −Y = left · +Y = right · +Z = up

This differs from standard ROS (+X forward, +Y left). Our robot faces −X, so the
right-hand rule gives left = Z × (−X) = **−Y**. Don't assume +Y = left.

`_applyUp('+Z')` applies `world.rotation.set(-PI/2, 0, 0)`, mapping:

| URDF axis | Three.js axis |
|-----------|--------------|
| X         | X            |
| Y         | −Z           |
| Z         | +Y (up)      |

Default camera is placed along direction `(-1, 0.7, 1)` from the robot center,
putting it at roughly URDF (−5.5, −5.5, +3.5) — the front-left corner viewed from above.

**Before editing any joint position, verify screen placement with this snippet:**

```python
import numpy as np

def screen_x(urdf_x, urdf_y, urdf_z):
    """Positive = screen right, negative = screen left."""
    tx, ty, tz = urdf_x, urdf_z, -urdf_y   # R_x(-PI/2)
    right = np.array([1, 0, 1]) / np.sqrt(2)
    return float(np.dot([tx, ty, tz], right))

# Example
print(screen_x(-0.010, -0.090, -0.002))  # wheel_left → negative = screen left ✓
print(screen_x(-0.010,  0.090, -0.002))  # wheel_right → positive = screen right ✓
```

## robot-car URDF assembly

The URDF is assembled in the browser from part files in `example/public/robots/robot-car/parts/`.
`robot-car.urdf` is not used — the browser fetches a `.parts.json` manifest, loads all part files
in parallel, and assembles them via a blob URL. Edit the source parts directly:

| File | Contents |
|------|----------|
| `00-intro.xml` | `<robot>` wrapper + `base_link` |
| `01-chassis.xml` | Chassis link + joint |
| `02-motors.xml` | Left + right TT motors |
| `03-wheels.xml` | Left + right drive wheels |
| `04-caster.xml` | Caster plate, fork, wheel |
| `05-motor-driver.xml` | L298N |
| `06-power.xml` | Battery box + power bank |
| `07-esp32cam.xml` | ESP32-CAM-MB (PRIMARY MCU) |

## Resizable panel

W3C APG Window Splitter pattern. `role="separator"` + `tabindex="0"`.
Min 260 px, max min(700 px, 60 vw), default 380 px. `localStorage` key `urdf-panel-w`.
**Do not** persist open/closed state. Handle hidden on mobile (`max-width: 768px`).

## Sidebar UI conventions

Desktop developer-tool genre (VS Code, Chrome DevTools). Compact, information-dense, small fonts.
Controls must be tappable on touch without looking like a mobile app. Apple HIG 44pt targets
are too large for a secondary tool panel.

CSS variables live in `:root`. Scrollable region is `.sidebar-scroll` (flex: 1, overflow-y: auto).

## Gesture mode (`example/gesture.ts`)

Hand-tracking via MediaPipe `@mediapipe/tasks-vision` loaded from CDN at runtime.
The module is lazy-loaded on first button click (separate Vite chunk, no bundle impact).

### Gesture dispatch — one gesture, one job

| Gesture | MediaPipe label | Action |
|---------|----------------|--------|
| Closed Fist | `Closed_Fist` | Orbit camera (wrist lm[0] delta) |
| Pointing Up | `Pointing_Up` **or landmark check** | Hover highlight + dwell select (fingertip lm[8]). Works at any angle — ML label catches upward, landmark geometry (index extended + others curled) catches sideways/diagonal. |
| Open Palm | `Open_Palm` | Reset all joints (hold 1 s) |
| Any + joint selected | any | Wrist roll drives joint angle |
| Two hands | any | Pinch/spread zoom |

Each branch clears the state owned by every other branch.

### MediaPipe configuration

```typescript
minHandDetectionConfidence: 0.7   // 0.5 default causes false detections in cluttered scenes
minHandPresenceConfidence:  0.5
minTrackingConfidence:      0.5   // keep lower so fast moves don't drop tracking
// NO delegate: 'GPU' — gesture subgraph has CPU-only ops (mediapipe issue #4712)
```

### Recognition loop pattern

```typescript
// Guard: camera ~30fps, rAF ~60fps — skip duplicate frames
if (video.currentTime !== this._lastVideoTime) {
    this._lastVideoTime = video.currentTime;
    const result = recognizer.recognizeForVideo(video, performance.now());
    // performance.now() is correct for live camera (not video.currentTime * 1000)
}
```

### Signal smoothing: One Euro Filter

Raw MediaPipe landmarks jitter 5–10mm at arm-extended distances. We use the **One Euro Filter**
(Casiez et al., CHI 2012) on the wrist position before computing orbit deltas. Unlike simple EMA,
it adapts cutoff frequency to velocity: stationary hand → heavy smoothing; fast movement → low
latency. Two parameters: `f_min` (1.0 Hz) controls smoothing when still; `β` (0.007) controls
how quickly the filter opens up with speed.

### Key landmark indices

| Index | Landmark |
|-------|---------|
| 0 | Wrist |
| 5 | Index MCP |
| 8 | Index fingertip |
| 12 | Middle fingertip |

## Gesture design constraints

Max 5 gestures (working memory limit, IJHCI 2024). Do not add more without removing one.
Prefer iconic gestures (fist=grab, palm=release) over cultural emblems.

Dwell selection: 800ms (research sweet spot 700-900ms). 30px movement resets the timer.

Wrist roll comfortable range: +/-30-40 degrees from neutral. Delta-based mapping avoids limits.

**Known fatigue issue:** Open_Palm hold for 1s is a sustained static pose. If complaints arise,
replace with a dynamic gesture (e.g., fist-to-open snap).
